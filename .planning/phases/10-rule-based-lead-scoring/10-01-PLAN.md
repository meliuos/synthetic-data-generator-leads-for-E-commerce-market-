<objective>
Implement Phase 10: Rule-Based Lead Scoring Engine in one plan. Build the Python scoring module
(no external deps), a comprehensive unit test suite, and a ClickHouse view that mirrors the same
rule logic for queryable scores across all sessions.

Deliverables:
1. src/scoring/rules.py — table-driven engine; SessionFeatures, LeadScore, _RULES, score_session()
2. src/scoring/__init__.py — package re-exports
3. tests/test_scoring_rules.py — full coverage across all six rules, clamping, tiers, batch API
4. infra/clickhouse/sql/005_phase10_lead_scoring.sql — analytics.lead_scores_rule_based view
5. scripts/smoke_phase10.sql — six validation queries
6. Makefile schema-phase10 + smoke-test-phase10 targets
</objective>

<context>
analytics.session_features (Phase 9) is available and provides the feature vector.
The Python module must be importable without ClickHouse or ML libraries — it runs in unit tests
and in future Phase 12 dashboard code that may run outside Docker.

Rule table design constraint: rules are data, not code. Each entry is (name, delta, predicate).
Adding or retuning a rule requires only an edit to _RULES, and both the Python and ClickHouse
representations must stay in sync.

max_scroll_pct NULL semantics: NULL means no scroll events fired, not "scrolled to 0%". The
scroll_engagement rule must NOT fire when max_scroll_pct is None (Python) or NULL (ClickHouse).
In ClickHouse this is handled by ifNull(max_scroll_pct, 0.0) which maps NULL to 0.0, safely
below the 70.0 threshold.

Bouncer semantics: penalises single-page sessions with zero e-commerce signals. Retailrocket
sessions have page_views=0 (not 1), so the bouncer predicate naturally never fires for that
source — no special-casing needed.
</context>

<constraints>
- No external Python dependencies in src/scoring/
- Six rules only — no additions without a corresponding ClickHouse view update
- Tier constants must be module-level (TIER_HOT_MIN, TIER_WARM_MIN) for import by Phase 12
- ClickHouse view is idempotent (CREATE OR REPLACE VIEW)
- Score is always clamped to [0, 100] — no negative scores in output
- rule_contributions in LeadScore must omit non-fired rules (not include them with delta 0)
</constraints>
