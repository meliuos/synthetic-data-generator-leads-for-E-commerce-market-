  <action>
Create a new executable bash script at `scripts/smoke-test-v11.sh`. Mirror the structure of `scripts/smoke-test.sh` (v1.0 reference) — same shebang, same `set -euo pipefail`, same env-var overrides, same `$COMPOSE exec -T redpanda rpk topic produce` producer pattern, same polling-with-deadline read pattern.

Script responsibilities:

1. **Prelude** — set `COMPOSE`, `CLICKHOUSE_USER`, `CLICKHOUSE_PASSWORD`, `TOPIC`, and a unique `SESSION_ID="ecom-smoke-$(date +%s)"`. Derive a deterministic `ORDER_ID="ord-${SESSION_ID}"` for dedup assertions. `chmod +x` the file.

2. **Produce 4 events to the Kafka topic `lead-events`** (topic name must match v1.0). Use `rpk topic produce -f '%v\n'` exactly as v1.0 does. Each event is a single-line JSON document:

   **Event A — `product_view` with FLAT shape (top-level e-commerce fields, no `properties` wrapper):**
   ```
   {
     "event_id":"<uuid-like string>",
     "event_time":"<ISO-8601 UTC>",
     "event_type":"product_view",
     "type":"product_view",
     "page_url":"https://example.com/product/SKU-100",
     "session_id":"$SESSION_ID",
     "anonymous_user_id":"anon_ecom_smoke",
     "device_type":"desktop",
     "product_id":"SKU-100",
     "category":"electronics/headphones",
     "price":99.99,
     "event_payload":"{\"source\":\"smoke-v11\",\"shape\":\"flat\"}"
   }
   ```

   **Event B — `add_to_cart` with NESTED `properties` JSON (properties-only shape):**
   ```
   {
     "event_id":"<uuid-like string>",
     "event_time":"<ISO-8601 UTC>",
     "event_type":"add_to_cart",
     "type":"add_to_cart",
     "page_url":"https://example.com/product/SKU-200",
     "session_id":"$SESSION_ID",
     "anonymous_user_id":"anon_ecom_smoke",
     "device_type":"desktop",
     "properties":{
       "product_id":"SKU-200",
       "quantity":3,
       "price":19.50
     },
     "event_payload":"{\"source\":\"smoke-v11\",\"shape\":\"nested\"}"
   }
   ```

   **Event C — `purchase` with nested `properties.products[]` (the one that must fan out per-line-item):**
   ```
   {
     "event_id":"<uuid-like string>",
     "event_time":"<ISO-8601 UTC>",
     "event_type":"purchase",
     "type":"purchase",
     "page_url":"https://example.com/checkout/thankyou",
     "session_id":"$SESSION_ID",
     "anonymous_user_id":"anon_ecom_smoke",
     "device_type":"desktop",
     "properties":{
       "order_id":"$ORDER_ID",
       "total":139.49,
       "revenue":139.49,
       "currency":"USD",
       "products":[
         {"product_id":"SKU-100","name":"Headphones","category":"electronics/headphones","price":99.99,"quantity":1,"position":1},
         {"product_id":"SKU-200","name":"Cable","category":"electronics/cables","price":19.50,"quantity":2,"position":2}
       ]
     },
     "event_payload":"{\"source\":\"smoke-v11\",\"shape\":\"purchase\"}"
   }
   ```

   **Event D — duplicate of Event C** (same `order_id`, different `event_id`, different `event_time`; proves `ReplacingMergeTree` server-side dedup). Use `event_time` a few milliseconds later than Event C so the later row wins under `ReplacingMergeTree(event_time)` semantics.

   JSON encoding note: bash here-docs with embedded quotes are error-prone for nested JSON. The script may build each JSON payload via `cat <<'EOF'` with variable substitution in a second pass, or pipe through `jq -c` if `jq` is available. Pick whichever approach keeps the file readable and emits ONE JSON object per line when piped to `rpk topic produce -f '%v\n'`.

3. **Bounded poll loop — verification section (5 assertion blocks)**

   Use the same deadline pattern as v1.0 (`deadline=$((SECONDS + 10))` — give the e-commerce MVs a little more time than the single-table v1.0 test). Each assertion either passes or fails the script. On failure, print the diagnostic row count / row contents so a developer can debug.

   **Assertion 1 — SCHEMA-01 coverage: all 8 new columns populated at least once across test events.**
   Query `analytics.click_events` filtered by `session_id = '$SESSION_ID'`, assert:
   ```sql
   SELECT
     countIf(product_id IS NOT NULL)    AS c_product_id,
     countIf(category IS NOT NULL)      AS c_category,
     countIf(price IS NOT NULL)         AS c_price,
     countIf(quantity IS NOT NULL)      AS c_quantity,
     countIf(order_id IS NOT NULL)      AS c_order_id,
     countIf(cart_value IS NOT NULL)    AS c_cart_value,
     countIf(search_query IS NOT NULL)  AS c_search_query,
     countIf(results_count IS NOT NULL) AS c_results_count
   FROM analytics.click_events
   WHERE session_id = '$SESSION_ID'
   ```
   Expected: `c_product_id >= 2`, `c_category >= 2`, `c_price >= 2`, `c_quantity >= 1`, `c_order_id >= 2` (from the two purchase rows — note the raw purchase events also have `order_id`). `cart_value`, `search_query`, `results_count` may be 0 in this smoke test (not exercised) — do NOT assert on them.

   **Assertion 2 — SCHEMA-02 flat extraction: Event A populated from flat top-level keys.**
   ```sql
   SELECT product_id, category, price
   FROM analytics.click_events
   WHERE session_id='$SESSION_ID' AND event_type='product_view'
   ```
   Expected: one row with `product_id='SKU-100'`, `category='electronics/headphones'`, `price` castable to `99.99`.

   **Assertion 3 — SCHEMA-02 nested extraction: Event B populated from `properties` fallback.**
   ```sql
   SELECT product_id, quantity, price
   FROM analytics.click_events
   WHERE session_id='$SESSION_ID' AND event_type='add_to_cart'
   ```
   Expected: one row with `product_id='SKU-200'`, `quantity=3`, `price` castable to `19.50`.

   **Assertion 4 — Per-line-item secondary MV (replaces "products[] ARRAY JOIN projection"):**
   ```sql
   SELECT count(), groupArray(product_id)
   FROM analytics.purchase_items
   WHERE order_id = '$ORDER_ID'
   ```
   Expected: `count() = 4` (two duplicate purchase events × two line items each = 4 rows). The sibling table does NOT dedup — that is the `orders` table's job.
   Sanity sub-check: `groupArray(product_id)` contains both `'SKU-100'` and `'SKU-200'`.

   **Assertion 5 — Server-side `order_id` dedup (replaces "ReplacingMergeTree projection"):**
   ```sql
   SELECT count() FROM analytics.orders FINAL WHERE order_id = '$ORDER_ID'
   ```
   Expected: `count() = 1`. Must use `FINAL` — without `FINAL`, the background merge hasn't necessarily fired yet and count can be 2 (both inserted rows). `FINAL` forces query-time dedup per RESEARCH.md §5.

   **Assertion 6 — GA4 alias view exposes renamed columns:**
   ```sql
   SELECT item_id, item_category, transaction_id, search_term
   FROM analytics.click_events_ga4
   WHERE session_id = '$SESSION_ID' AND event_type = 'product_view'
   ```
   Expected: exactly one row, `item_id='SKU-100'`, `item_category='electronics/headphones'`, `transaction_id` NULL (not a purchase row), `search_term` NULL. The assertion is structural — the script checks that the SELECT succeeds with these exact column names (proves the view has them) AND that `item_id` is non-empty.

4. **PASS / FAIL output**

   On success, print `PASS: v1.1 e-commerce schema smoke test (session_id=$SESSION_ID, order_id=$ORDER_ID)` and `exit 0`. On any assertion failure, print the exact query, expected value, observed value, and `exit 1`.

5. **Cleanup (optional, not required)**

   The script intentionally does NOT delete the test rows. Tests accumulate by `SESSION_ID` which is unique-per-run; they do not poison future runs. If the developer wants to clean up they can `TRUNCATE` tables manually.

Make the file executable: include `#!/usr/bin/env bash` shebang; the plan executor will `chmod +x` it as part of the commit.
