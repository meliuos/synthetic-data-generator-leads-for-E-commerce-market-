<objective>
Implement Phase 11: ML Lead Scoring Model. Train a LightGBM binary classifier on the
Retailrocket session_features corpus, export it, and wire up a batch scoring pipeline
that writes ML predictions to analytics.lead_scores_ml alongside the existing rule-based
scores from Phase 10.

Deliverables:
1. notebooks/lead_scoring_model.ipynb — training notebook with CV, baseline comparison, model card
2. src/scoring/ml_scorer.py — lazy-loading MLScorer wrapper (no import-time ML deps)
3. scripts/score_sessions.py — batch CLI: pull sessions → score → insert to ClickHouse
4. infra/clickhouse/sql/006_ml_scores.sql — analytics.lead_scores_ml table schema
5. scripts/smoke_phase11.sql — four validation queries
6. requirements-ml.txt — ML dependency pinning
7. Makefile targets: schema-phase11, smoke-test-phase11, ml-setup, score-sessions
</objective>

<context>
analytics.session_features (Phase 9) supplies all features. analytics.lead_scores_rule_based
(Phase 10) supplies the baseline to beat. The ML model scores the same feature vector so
Phase 12 can join both scores for a unified leads table.

Class imbalance: Retailrocket has ~0.82% conversion rate. scale_pos_weight = neg/pos
handles this without resampling, which avoids leaking positives across folds.

Feature gap: Retailrocket has no scroll events (max_scroll_pct always NaN) and no search
events (search_count always 0). LightGBM handles NaN natively; do NOT impute to 0 or the
signal is corrupted. search_count=0 for all rows is a zero-variance feature for Retailrocket
training — LightGBM will assign it zero importance, which is correct.

Model storage: models/lead_scorer_lgbm.pkl is gitignored (binary artifact). The models/
directory is tracked via .gitkeep. Regenerate with: make ml-setup && jupyter nbconvert
--to notebook --execute notebooks/lead_scoring_model.ipynb && make score-sessions.
</context>
